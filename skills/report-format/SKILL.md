# 標準化報告模板

## A. 程式碼審查報告（code-reviewer 用）

```
## 程式碼審查報告

**審查對象**：[檔案路徑或範圍]
**審查時間**：[當前時間]
**整體評分**：🟢 良好 / 🟡 需改進 / 🔴 有重大問題

---

### 重大問題（必須修正）
[問題描述，附上檔案行號]
**建議**：[具體修正方式]

### 改進建議（應該修正）
[問題描述，附上檔案行號]
**建議**：[具體修正方式]

### 優點（值得保留的好實踐）
[指出寫得好的部分，鼓勵好的習慣]

### 總結
[整體評語，3-5 句話]
```

## B. 工作報告（code-writer 用）

```
## 工作報告

### 任務摘要
[一句話描述本次任務]

### 已完成項目
- [檔案:行號] 變更描述

### 未完成項目（需人工確認）
- [原因說明]

### 建議下一步
重新執行 code-reviewer 驗證結果。
```

## C. Retro 記錄（retro 用）

```markdown
---

## [YYYY-MM-DD] Retro — [審查對象]

### 問題清單
| # | 嚴重度 | 問題摘要 | 根因分類 |
|---|--------|---------|---------|
| 1 | 🔴 | ... | 知識不足 / 疏忽 / 架構限制 / ... |

### 教訓
1. **[模式名稱]**
   - 根因：...
   - 預防：...

### 行動項目
- [ ] [具體可執行的改善動作]
```

## D. 任務審查報告（task-reviewer 用）

```
## 任務審查報告

**審查對象**：task.md（新增任務 N 項）
**審查時間**：[當前日期]
**整體評分**：🟢 可直接開工 / 🟡 建議補強 / 🔴 需重寫或拆分

---

### 任務 1：[任務標題]

**清楚程度**：🟢 / 🟡 / 🔴
[評語]

**拆分建議**：
- [是否需要拆分？拆成哪幾個子任務？]

**技術限制**：
- [列出需要注意的技術限制、前置條件]

**風險提醒**：
- [可能的 regression、資料風險、部署影響]

**已直接修改 task.md**：是 / 否
[若有修改，簡述修改內容]

---

### 總結
[整體評語，指出哪些任務可直接交給 code-writer，哪些需要先跟使用者對齊]
```

## E. 任務完成度驗證報告（task-verifier 用）

```
## 任務完成度驗證報告

**任務**：[任務編號 — 任務標題]
**驗證時間**：[當前日期]
**判定**：🟢 通過 / 🟡 有條件通過 / 🔴 未通過

---

### 子任務完成度
| # | 子任務 | 狀態 | 證據 |
|---|--------|------|------|
| 22-1 | [描述] | ✅ 完成 / ❌ 遺漏 / ⏭️ 刻意跳過 | [diff 行號或 grep 結果] |

### DoD 逐條驗證
| # | 驗收標準 | 通過 | 證據 |
|---|---------|------|------|
| 1 | [標準描述] | ✅ / ❌ / 🔍 需人工驗證 | [具體證據] |

### Scope 偏移
- [列出 diff 中有改但 task 沒提到的檔案/區域]
- [列出 task 提到但 diff 中沒改的檔案/區域]
- [「不做的事」違反檢查結果]

### 技術限制遵守
- [逐一列出 task 中標註的限制，確認是否遵守]

### 總結
[整體評語：完成度百分比、遺漏項目、需要人工確認的項目]
```
