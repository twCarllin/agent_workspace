# 根因分類參考

## 分類表

| 分類 | 說明 | 典型預防措施 |
|------|------|-------------|
| 知識不足 | 不熟悉語言/框架特性 | 文件補充、pair programming |
| 疏忽 | 知道但忘了 | checklist、linter rule |
| 複製貼上 | 從他處搬來未調整 | 重構為共用函數 |
| 架構限制 | 現有架構難以正確處理 | 技術債清理 |
| 需求模糊 | 規格不明導致實作偏差 | 需求確認流程 |
| 趕工 | 時間壓力下跳過品質步驟 | 估時調整、scope 縮減 |
| 時序耦合 | 多個延遲/非同步機制疊加產生看不見的競態 | 畫時序圖、減少 state store 數量 |
| 驗證盲點 | 驗收只看「有沒有炸」沒驗「數字是否正確」 | 要求具體可對帳的測項 |

## 分析框架

對每個問題，逐一思考：
1. **為什麼會寫出這段有問題的程式碼？**（趕工、不熟悉 API、缺乏 context、複製貼上…）
2. **是個案還是系統性問題？**（只出現一次 vs 多處重複同樣錯誤）
3. **現有的流程/工具有沒有辦法提前攔截？**（linter、type check、test、code review checklist）
